# CLAUDE.md — Operating Rules для подпроекта AIM/SSA

## 1. Расположение и стек

- **Путь**: `~/Desktop/AIM/SSA/` — монорепозиторий AIM.
- **Backend**: Rust (axum) → `ssa-engine/`.
- **Frontend**: Phoenix LiveView → `ssa-ui/`.
- **LLM-интеграция**: только через `~/Desktop/AIM/llm.py` (единый шлюз, без прямых вызовов OpenAI/Anthropic).
- **Тестирование**: `cargo test` (Rust), `mix test` (Elixir).
- **Валидация паттернов**: собственный набор ~200 CBC-золотых кейсов (`test/fixtures/gold_set.json`).

## 2. Стартап и проверка консистентности

При начале работы сессии:

1. Прочитать (если не загружены в контекст) файлы:
   - `CONCEPT.md` — определение SSA, 5-зонная методология.
   - `STATE.md` — текущее состояние движка, известные баги.
   - `OPEN_PROBLEMS.md` — нерешённые вопросы (например, пороги для беременных, флаги артефактов).
   - `EVIDENCE.md` — ссылки на источники для каждого порога и комбинации.
   - `ARCHITECTURE.md` — слои data → engine → API → UI.
   - `PARAMETERS.md` — список 28 параметров с единицами и референсами.
   - `PATTERNS.md` — описание 15 парных и 15 тройных синдромальных паттернов.
   - `METRICS.md` — метрики качества SSA (top-1/top-3 accuracy, red-flag-miss-rate и т.д.).
   - `DEPLOYMENT.md` — инструкции по сборке и запуску.

2. Проверить консистентность между файлами:
   - Каждый параметр из `PARAMETERS.md` присутствует в `sources/parameter_*.md`.
   - Все паттерны из `PATTERNS.md` имеют соответствующий JSON в `patterns/*.json`.
   - `ranges.json` (в `sources/ranges.json`) синхронизирован с `sources/parameter_*.md`.
   - `patterns.json` (в `patterns/patterns.json`) синхронизирован с `patterns/*.md`.
   - Ни один из 9 файлов не содержит взаимоисключающих утверждений (например, разные референсные интервалы для Hb у взрослых мужчин).

3. Если обнаружена несогласованность — зафиксировать в `STATE.md` как баг и не вносить изменения, пока не получено разрешение.

## 3. Источники истины

- **Референсные интервалы**: `sources/parameter_*.md` → `sources/ranges.json`.
  - Каждый параметр имеет отдельный `.md`-файл, где указаны:
    - Источник (ICSH 2014, NHANES, Wintrobe, BCSH, WHO).
    - Популяционные нормы (мужчины/женщины, взрослые/дети, беременность, высокогорье).
    - Критические клинические пороги (L2, H2).
    - Обоснование выбора границ L1/L0/H1/H2.
  - JSON-версия (`ranges.json`) — плоский файл, генерируется скриптом `scripts/build_ranges.py` из `.md`.
  - **Никогда не редактировать `ranges.json` напрямую** — сначала править `.md`, потом запускать скрипт.

- **Паттерны**: `patterns/*.md` → `patterns/patterns.json`.
  - Каждый паттерн (парный или тройной) описан в `.md` с указанием:
    - Диагностических критериев (какие зоны каких параметров входят).
    - Дифференциального ряда.
    - Доказательной базы (ссылки: Bessman 1983, Mentzer, Zahorec, и т.д.).
  - JSON-версия (`patterns.json`) генерируется скриптом `scripts/build_patterns.py`.
  - Править `.md`, затем запускать скрипт.

## 4. Процесс внесения изменений

1. **Изменение референсных интервалов**:
   - Отредактировать `sources/parameter_<name>.md` с указанием нового источника в `EVIDENCE.md`.
   - Запустить `cargo run --bin build_ranges` (или `./scripts/build_ranges.py`).
   - Убедиться, что `ranges.json` перестроился корректно (diff).
   - Запустить тесты: `cargo test`.
   - Если тесты падают — не коммитить.

2. **Изменение паттернов**:
   - Отредактировать `patterns/<pattern_name>.md`.
   - Запустить `./scripts/build_patterns.py`.
   - Проверить `patterns/patterns.json`.
   - Запустить тесты (включая gold set validation): `cargo test && mix test`.
   - Проверить, что accuracy на gold set не снизилась (сравнить с `METRICS.md` baseline).

3. **Изменение логики движка** (zonal mapper, pattern matcher):
   - Писать код в `src/` (Rust).
   - Добавить unit-тесты для новых функций.
   - Добавить интеграционный тест с новым синдромальным паттерном (в `tests/`).
   - Запустить `cargo clippy`, `cargo fmt`.
   - Прогнать полный набор тестов.

4. **Изменение UI** (Phoenix):
   - Править `lib/ssa_ui/`.
   - После изменений — `mix test`.
   - Визуально проверить в dev (доступен по localhost:4000).

## 5. Интеграция с LLM

- Единственный разрешённый путь: `~/Desktop/AIM/llm.py`.
- **Не вызывать API OpenAI/Anthropic напрямую** (ни из Rust, ни из Elixir).
- При необходимости уточнения референсов или дифференциальной диагностики — формировать запрос через `llm.py` с указанием:
  - Контекста (текущий паттерн, параметры, зоны).
  - Какой именно вывод требуется (например, «список нозологий для синдрома ANEMIA_MICROCYTIC_SET с RDW↑»).
  - Уровня доказательности (требуется указание источника).
- Ответ LLM должен быть записан в `EVIDENCE.md` с пометкой "LLM-suggested" и последующей верификацией.
- **Запрещено** использовать LLM для генерации референсных интервалов без записи в `EVIDENCE.md` и без проверки по первоисточникам.

## 6. Референсные интервалы — жёсткая фиксация

- Границы зон (L2, L1, L0, H1, H2) считаются фиксированными после утверждения. Изменение возможно только при:
  1. Появлении нового крупного популяционного исследования (ICSH, WHO, NHANES).
  2. Клиническом консенсусе (BCSH, ASH).
  3. Обязательном обновлении `EVIDENCE.md` с цитатой из источника.
- Любое изменение порогов без регистрации в `EVIDENCE.md` является критической ошибкой и должно быть откачено.

## 7. Тестирование

- **Rust backend**:
  - `cargo test` — все unit и integration тесты.
  - Отдельный тест `test_zonal_mapper` — проверяет цифровизацию каждого параметра для всех границ.
  - `test_pattern_matcher` — проверяет, что для известных синдромов (например, панцитопения) возвращается правильный топ-3.
  - `test_gold_set` — accuracy на ~200 CBC (метрики: top-1 ≥ 0.85, top-3 ≥ 0.95, red-flag-miss-rate = 0).
  - `test_artifact_handling` — проверка обработки псевдотромбоцитопении, холодовых агглютининов и т.д.

- **Elixir frontend**:
  - `mix test` — все тесты Phoenix.
  - `test_ui/ssa_live_test.exs` — тестирование отображения ранжированного списка синдромов.

- **Gold set**:
  - Находится в `test/fixtures/gold_set.json`.
  - Каждый кейс содержит: исходные CBC-значения, ожидаемый синдромный паттерн (основной + дифференциальный ряд).
  - При добавлении нового паттерна — обязательно добавить минимум 5 кейсов в gold set.

## 9. Связь с DiffDiagnosis

- SSA — первый этап: CBC → синдромы.
- Выход SSA (ранжированный список синдромов) передаётся в `DiffDiagnosis` (другой подпроект) через общий JSON-формат (`ssa_output.json`).
- Формат:
  ```json
  {
    "syndromes": [
      {"name": "ANEMIA_MICROCYTIC_SET", "score": 0.95, "zones": {"HGB": "L2", "MCV": "L1", "MCH": "L2", "RDW": "H1"}},
      {"name": "ANEMIA_CHRONIC_INFLAMMATION", "score": 0.60, ...}
    ],
    "flags": ["red_flag_pancytopenia", "red_flag_tma"],
    "artifacts": ["cold_agglutinins_possible"]
  }
  ```
- **Не дублировать логику DiffDiagnosis внутри SSA.** SSA не ставит нозологии — только синдромы.
- При изменении формата вывода — синхронизироваться с DiffDiagnosis (`~/Desktop/AIM/DiffDiagnosis/`).

## 10. Запреты и ограничения

1. **Не изменять `ranges.json` или `patterns.json` вручную** — только через скрипты.
2. **Не добавлять новые параметры в CBC** без обсуждения (расширение списка 28 параметров затрагивает все комбинации и документацию).
3. **Не использовать LLM для генерации клинических рекомендаций** — SSA только помогает дифференциальной диагностике, рекомендации должны быть основаны на проверенных руководствах.
4. **Не удалять red-flag паттерны** (PANCYTOPENIA, THROMBOTIC_MICROANG, LEUKOERYTHROBLASTIC) — они обязательны к обнаружению.
5. **Не отключать проверку артефактов** — функция `detect_artefacts` должна быть всегда активна.

## 11. Порядок работы при ошибке

Если тесты падают или gold set accuracy падает ниже порога:

1. Откатить последнее изменение.
2. Проверить, не изменились ли границы зон (сравнить `ranges.json` с baseline в `docs/meta/03_population_refs.md`).
3. Проверить, не появились ли новые артефакты (например, новый анализатор даёт RDW-CV в другом диапазоне).
4. Записать в `STATE.md` описание проблемы и ссылку на коммит.
5. Не деплоить до исправления.
