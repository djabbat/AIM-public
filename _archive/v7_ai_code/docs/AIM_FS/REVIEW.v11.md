## CRITICAL ISSUES

Нет критических проблем, блокирующих MVP. Все предыдущие замечания (отсутствие DDL, таблица locks, UUID, concurrent propose, reject workflow, L10) исправлены в v10.

Единственное наблюдение (не блокирующее):

1. **Секция 4.2 (Блокировочный протокол)**: `BEGIN IMMEDIATE` для **каждой** модифицирующей транзакции захватывает эксклюзивную блокировку всей БД. При batch‑propose это может увеличить latency, если БД содержит >10⁴ записей. В рамках заявленной нагрузки (<10 concurrent ops) это не критично, но для будущего multi‑tenant (Phase B) потребуется пересмотр. В MVP можно оставить, но стоит явно указать, что это сознательный компромисс.

## MINOR ISSUES

1. **Секция 3.1 (Frontmatter)**: В примере отсутствует поле `on_expire` (упомянуто в тексте как `“keep” | “deprecate”`). Нужно включить в пример для ясности.

2. **Секция 5.1 (DDL)**: Поле `decay_on_expire` в таблице `entities` имеет тип `TEXT DEFAULT 'keep'`, но в frontmatter используется `on_expire`. Несоответствие именования – следует унифицировать (например, `decay_on_expire` в обоих местах).

3. **Секция 8 (Conflict resolution)**: Указано, что при конфликте новая запись получает статус `disputed`. В таблице `entities` нет такого статуса (есть `pending`, `active`, `superseded`, `rejected`, `deprecated`, `expired`). Нужно либо добавить статус `disputed`, либо явно указать, что конфликт хранится отдельно (таблица disputes), а запись остаётся `pending` с блокировкой.

4. **Секция 10.1 (Idempotency)**: Описан механизм с `processing`/`done` и 5‑секундным таймаутом. При повторном запросе с тем же ключом возвращается `"conflict"`. Это корректно, но в тексте сказано: "клиент должен отправить с новым ключом". Лучше уточнить, что ошибка `"conflict"` означает "запрос уже обрабатывается, повторите с другим ключом", чтобы избежать неоднозначности.

5. **Секция 10.2 (Phoenix context)**: Упомянут heartbeat (ping каждые 5 сек). Не указано, что происходит, если Rust‑процесс не отвечает – spec говорит "три пропуска = перезапуск". Хорошо, но стоило бы упомянуть, что при перезапуске очищаются `processing` idempotency записи (указано в 10.1, но дублирование не помешает).

6. **Секция 12 (Преимущества)**: В столбце "Статус" для L10 написано `"MVP превосходит"` с пояснением. В таблице количество осей теперь 13, но в тексте выше написано 15. В заголовке таблицы 15 строк, из них 13 превосходят – это корректно, но требует внимания при чтении. Не ошибка, но можно унифицировать.

## STRONG POINTS

- **Детальная проработка атомарности**: использование `BEGIN IMMEDIATE` + optimistic locking + идемпотентность через таблицу. Для single‑user это избыточно, но закладывает основу для будущего multi‑tenant.
- **Полный DDL**: все таблицы и индексы явно объявлены, включая частичный индекс для sweeper. Это снимает неоднозначности при реализации.
- **Edge cases**: тщательно рассмотрены LLM‑мусор, null confidence, revert с проверкой depends_on, concurrent propose, sweeper vs user, GDPR delete.
- **Чёткое разделение MVP vs Roadmap**: каждая функция, отложенная на потом, явно маркирована (например, CAS, FTS5, каскадный decay, шифрование). Это предотвращает scope creep.
- **Исправление предыдущих замечаний**: все 6 critical issues из v9 закрыты; изменения задокументированы в changelog.

## SUGGESTED ADDITIONS / SIMPLIFICATIONS

1. **Упростить idempotency для MVP**: заменить таблицу `idempotency` на in‑memory `HashMap<Key, State>` с TTL (например, `tokio::sync::RwLock`). Это снизит сложность и latency (не нужно писать в БД). Таблицу можно оставить для Phase 2. Нагрузка ≤10 ops – in‑memory более чем достаточно.

2. **Добавить статус `disputed` в DDL** (см. Minor Issues #3) или явно описать, что конфликтная запись остаётся `pending` с блокировкой и ссылкой на dispute.

3. **Уточнить про `BEGIN IMMEDIATE` в batch‑propose** (секция 4.2): так как блокировка эксклюзивная, захват строк в порядке ID избыточен. Можно упростить: просто `BEGIN IMMEDIATE` без указания порядка строк. Это сэкономит время на реализацию.

4. **Добавить метрику "среднее время блокировки"** в SMART‑метрики (например, <10ms). Это позволит выявить проблемы с производительностью на раннем этапе.

5. **Рассмотреть возможность использования `r2d2` с одним соединением** для MVP – это исключит deadlock между соединениями при `BEGIN IMMEDIATE`. Для одного пользователя одно соединение достаточно.

## SUFFICIENCY VS CLAUDE

Спецификация убедительно демонстрирует превосходство над Claude Code по 13 из 15 заявленных осей:

- **L4 (полнотекстовый поиск)**: честно указано "уступает в MVP" (планируется FTS5 в Phase 2).
- **L6 (каскадный decay)**: указано "сравнимо в MVP" (каскад отложен до Phase 2).

Для остальных 13 осей приведены конкретные механизмы (таблицы, индексы, workflow), которые обеспечивают явное превосходство. Особенно выделяется L10 (conflict detection) – в spec показано, что Claude не имеет явного детектора, а AIM_FS реализует его уже в MVP.

**Вывод**: spec полностью соответствует заявленному требованию превосходства. Никаких скрытых дыр в обосновании нет.
