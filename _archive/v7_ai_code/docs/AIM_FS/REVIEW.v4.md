## CRITICAL ISSUES

1. **Идемпотентность Port‑вызовов (секция 10.2)** 
 Потеря сообщения at‑most‑once + отсутствие идемпотентности → повторный `approve` может выполниться второй раз, если первый успел записать, но не отправил ответ. Rust core должен проверять, что операция уже выполнена (по proposal_id) и возвращать предыдущий результат. Иначе возможны дубли или нарушение целостности.

2. **Migration: дедупликация по SHA без учёта схемы (секция 10.3)** 
 Два файла с одинаковым содержимым, но разной семантикой (feedback vs fact) будут объединены в одну запись с версиями, что неверно. Дедупликация должна проводиться с учётом `schema` и, вероятно, `scope`.

3. **Conflict resolution: таймаут оставляет обе записи `active` (секция 8, п.5)** 
 Если dispute не разрешён за `dispute_timeout_days`, обе contradicting записи остаются `active`. Это приводит к неконсистентному состоянию — система содержит противоречивые факты, на которые могут ссылаться другие записи. Требуется **block** хотя бы одной записи (например, более новой) или автоматический выбор на основе confidence/времени.

4. **Undo‑механизм для auto‑approve: отсутствует каскад (секция 4.3)** 
 Если отменяется запись, на которую ссылаются другие (`depends_on`), эти ссылки становятся битыми. Нужно либо запретить undo для записей с исходящими зависимостями, либо выполнить каскадное superseding (что сложно).

5. **CAS retention: не указан GC‑sweeper (секция 3.3)** 
 `retain_versions=3`, но нет описания, как и когда старые версии удаляются или архивируются. Если GC не запускается, CAS будет бесконечно расти. Нужно добавить фоновый sweeper (например, раз в сутки).

6. **Offline‑режим: бесконечное ожидание (секция 4.1)** 
 Если пользователь не вернулся, `auto_expire_at` не срабатывает, и proposal висит вечно. Нужен абсолютный лимит (например, 90 дней) или политика `auto‑expire` после превышения `max_inactivity_days` + разумного периода.

7. **Decay sweeper: топологическая сортировка может быть слишком дорогой (секция 7)** 
 Для 10⁵ записей с глубокими цепочками зависимостей построение и обход DAG займёт >200 ms, как заявлено. Лучше ограничить глубину обхода (например, до 3) или использовать batch‑обработку с инкрементальным обновлением.

8. **Глобальный поиск: невозможен full‑scan по body (секция 5.1)** 
 Заявлено, что глобальные запросы ограничены frontmatter. Пользователь не сможет найти факт по содержанию, не зная scope. Для MVP это приемлемо, но нужно явно указать в UX. Иначе — ложные ожидания.

9. **Event log: отсутствует блокировка при concurrent write (секция 3.4)** 
 Append‑only `jsonl` не атомарен под параллельной записью. Необходимо использовать per‑shard `.lock` и для events, иначе возможны перемешанные строки.

10. **Rust core: отсутствует idempotent key для операций (секция 10.1)** 
 API не содержит request_id. При перезапуске Port клиент не может отличить повтор. Нужно добавить опциональный `idempotency_key` (ULID) и проверку в Rust.

## MINOR ISSUES

- **Naming:** `identity.yaml` назван «изменяемый через proposal», но в тексте далее сказано «не является полностью immutable» — путает. Лучше: «mutated via protected proposal».
- **Секция 3.1:** `source: "user_command"` переименован из `user_explicit` — хорошо, но в таблице 11 (L5) всё ещё упоминается `user_command` как auto‑approve — это уже учтено.
- **Секция 4.1:** `max_inactivity_days` определён как per‑user preference, но не указан default в схеме `preferences.md`. Добавить.
- **Секция 6:** `_service/contradicts_index.json` — название не соответствует стилю (везде `jsonl` для логов, а тут `.json`). Лучше `contradicts_index.json` (так как это не лог, а индекс) — ок, но унифицировать.
- **Секция 7:** `on_dep_expire: "deprecate" | "keep"` — значение `"deprecate"` опечатка (deprec**a**te?), должно быть `"deprecate"` или `"deprecated"`? Лучше `"deprecate"`.
- **Секция 10.2:** `Port crash → orphan locks не возникает, т.к. flock автоматически снимается` — верно, но `flock` освобождается только при закрытии всех fd. Если процесс упадёт, ядро закроет fd → ок.
- **Секция 12 (Open Questions):** многие вопросы уже решены в v3, например, п.7 (encryption) и п.10 (concurrent writes). Рекомендую удалить или пометить `RESOLVED`.
- **Таблица 11 (L4):** заявлено «MVP уступает по скорости, но превосходит по гибкости» — гибкость не раскрыта. Лучше честно: «Phase 5 превосходит, MVP пока уступает».

## STRONG POINTS

- **Atomic commit с prepare/rename + per‑entity lock** — проверенное решение, корректное при условии одной файловой системы.
- **Materialized contradicts index** — разумный компромисс между производительностью и консистентностью (без блокировок графа при propose).
- **Каскадный decay с топологической сортировкой** — детальная проработка, хотя и сложная.
- **Encryption at‑rest с per‑user key и HSM** — промышленный уровень, упомянуты rotation и revocation.
- **SMART метрики** — конкретны и измеримы (кроме затрат на GC и consistency).

## SUGGESTED ADDITIONS / SIMPLIFICATIONS

1. **Упростить MVP:** отказаться от materialized contradicts index и каскадного decay. Проверку конфликтов делать при propose через сканирование links.jsonl в релевантных шардах (с LRU). Это сократит время реализации с 7 до 4 дней.
2. **Ввести политику абсолютного TTL для proposal** (например, 30 дней) независимо от offline. При истечении — `expired`, уведомление.
3. **Добавить idempotency_key** в API Rust core (`propose`, `approve`, `reject`). Phoenix генерирует ULID и повторяет при ошибке, Rust возвращает предыдущий результат.
4. **Dispute timeout:** по умолчанию `resolved_unresolved` не оставлять обе `active`. Лучше: автоматически выбирать запись с более высоким confidence / более старую, или помечать обе как `ambiguous`.
5. **Migration:** дедуплицировать по (SHA, schema). Два файла с разной схемой — разные записи.
6. **Cleanup orphan tmp:** при старте Rust core сканировать `_service/tmp/` и удалять файлы старше 1 часа.
7. **Ограничить глубину каскадного decay** до 3 уровней во избежание O(n²) обхода.
8. **Уточнить формат `proposal_v1`** в схеме — привести пример обязательных полей.
