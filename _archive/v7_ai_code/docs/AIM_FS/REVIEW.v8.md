## CRITICAL ISSUES

1. **Conflict detection race при `propose` (раздел 6, 8).** 
 Запрос `SELECT.. WHERE target_id = ? AND link_type = 'contradicts' AND status = 'active'` не блокирует найденные строки. Два одновременных propose могут не увидеть конфликт друг друга и оба создать `disputed` записи. 
 *Решение:* использовать `SELECT.. FOR UPDATE` на целевой строке (или на `target_id` в таблице `links`), либо уникальный индекс на `(source_id, target_id, link_type)` и обрабатывать ошибку вставки как конфликт.

2. **Lazy decay не гарантирует атомарность (раздел 7).** 
 `entity.rs::get` меняет статус на `expired` при проверке `expires_at`. Если `get` выполняется вне транзакции, возможна гонка: два вызова одновременно увидят просрочку и оба попытаются обновить статус, один перезапишет другой, и событие изменения может потеряться. 
 *Решение:* не менять статус при чтении. Вместо этого возвращать флаг `expired` и пересчитывать статус только при записи (или в фоновом sweeper). Либо явно документировать, что `get` всегда вызывается в транзакции с `FOR UPDATE`.

3. **Отсутствие Phoenix UI в MVP (раздел 10.2 и план).** 
 Заявленный срок 7-10 дней full-stack, но Plan выделяет 7 дней только на Rust core, без LiveView. Для доктора approval queue бесполезна без интерфейса. 
 *Требование:* либо включить минимальный Phoenix LiveView inbox (просмотр pending + accept/reject) в Phase 1, либо явно указать, что MVP — это CLI/API, а UI — Phase 2. Последнее противоречит задаче.

4. **Избыточность offline-режима (раздел 4.1).** 
 Логика `max_inactivity_days` + `max_absolute_pending_days` + продление таймера создаёт множество угловых случаев (как считать дни, если пользователь входит не каждый день?). Для одного доктора MVP достаточно простого: `auto_expire` через 7 дней, а при входе показывать просроченные. Усложнение неоправданно.

5. **Undo-механизм не детализирован (раздел 4.3).** 
 `undo_window_hours: 24` предполагает, что можно откатить auto-approved запись. Не описано, как предотвратить каскадные эффекты, если запись уже использовалась для решений. Для MVP безопаснее не реализовывать undo, а требовать создание proposal на изменение.

## MINOR ISSUES

- **Опечатка `on_dep_expire: "deprecate"`** (раздел 3.1). Должно быть `"deprecate"` → `"deprecate"`. Слово `deprecate` не существует. Исправить на `"deprecate"` или `"deprecate"`.
- **Connection pool из 10 соединений** (раздел 5.1) — избыточно для одного пользователя с ≤10 concurrent ops. Можно ограничиться 2-3 соединениями, упростив конфигурацию.
- **Watchdog с max transaction time 5s** (раздел 4.2) — для MVP отдельный процесс излишен. SQLite сам откатывает транзакции по таймауту при блокировках. Убрать или отложить.
- **Валидация confidence для system записей** (раздел 11.2): не указано, какие именно записи считать system. Рекомендуется чёткое поле `agent: "system"` в frontmatter.
- **Партиционирование SQLite** (раздел 5.1) — в MVP одна БД, упоминание шардинга в Phase 2. Ок, но фраза «для масштабирования в Phase 2 – шардинг по user_id» может сбить с толку. Лучше перенести в Roadmap.

## SUGGESTED ADDITIONS / SIMPLIFICATIONS

1. **Упростить offline-режим:** удалить `max_absolute_pending_days` и продление таймера. Оставить только `max_inactivity_days` (по умолч. 30) — если пользователь не входил дольше, то при входе все pending записи считаются просроченными и попадают в inbox с пометкой `expired_offline`. Логика становится тривиальной.

2. **Заменить lazy decay на sweeper:** реализовать простой фоновый процесс (Tokio task) каждые 60 секунд, который обновляет статус `active` → `expired` для просроченных записей. Это атомарно и не требует транзакций при чтении. Для MVP объём данных мал, нагрузка ничтожна.

3. **Убрать undo-механизм из MVP.** Вместо него — кнопка «предложить отмену» в inbox, которая создаёт proposal с типом `revert`. Это не требует дополнительной логики, proposal проходит обычный approval.

4. **Добавить FTS5 в core MVP** — хотя бы на заголовках и теле файлов (индексировать при approve). Для доктора поиск по тексту записей о пациентах critical. FTS5 встроен в SQLite, реализация в Rust — несколько часов. Без этого превосходство над Claude по L4 недоказано.

5. **Уточнить conflict detection:** предложить атомарную проверку через блокировку целевой сущности: `SELECT.. FROM entities WHERE id = ? FOR UPDATE` (или замок на `target_id` в таблице `locks`). Это решает race.

## SUFFICIENCY VS CLAUDE

- **Превосходит по 13/15 осям** — L1-L3, L5, L7-L15. 
- **Уступает по L4 (поиск)** — MVP только SQL WHERE, без полнотекстового поиска по телу. Для доктора это может быть критично (поиск по диагнозам, заметкам). Рекомендуется FTS5 в MVP. 
- **Сравнимо по L6 (decay)** — lazy decay аналогичен verify-on-use Claude, но без каскадного. Для одного пользователя разница несущественна. 

Если не добавить FTS5, заявленное превосходство по L4 остаётся на уровне обещания.
