## CRITICAL ISSUES

1. **Multi-tenant изоляция (секции 9, 10.1, 5.1)**: Единая SQLite БД `aim_fs.db` для всех пользователей. Даже с namespace `users/<user_id>/` в таблице `entities`, не указана штатная row-level security – отсутствует обязательное поле `tenant_id` во всех таблицах и его проверка в каждом запросе. При ошибке в SQL (например, забыли WHERE user_id) произойдёт утечка данных между пользователями. Шифрование PII at-rest не защищает метаданные (scope, links, tags), которые хранятся открытыми в SQLite и могут раскрыть связи между пациентами.

2. **Масштабирование SQLite под 100 concurrent users (секция 5, 10.1)**: WAL позволяет параллельные чтения, но одна транзакция записи блокирует всю БД. При 100 конкурентных propose/approve ожидаемая задержка >500ms (p95). Нет упоминания о пуле соединений (rusqlite по умолчанию одно соединение). Нет оценки, сколько write throughput выдержит SQLite до того, как станет bottleneck-ом. Для заявленных 100 ops/s – требуется бенчмарк.

3. **Idempotency race при timeout (секция 10.1)**: Если операция не завершилась за 30 секунд, строка `idempotency` удаляется, и клиент может ретраить. Но операция могла уже частично выполниться (коммит не прошёл до обновления idempotency) – нет гарантии, что повторный запрос не создаст дубликат. Атомарность не обеспечена, так как обновление idempotency и запись данных – в одной транзакции, но удаление строки происходит после транзакции (в рамках `gc.rs`). Нужно либо не удалять, а помечать как `failed`, либо использовать механизм блокировки до момента, пока предыдущая транзакция не завершится.

4. **Decay sweeper и concurrent update (секция 7)**: Sweeper захватывает строки в порядке возрастания ID с `SELECT.. FOR UPDATE`. Однако если пользователь одновременно обновляет ту же сущность (например, меняет TTL), возможен deadlock или потеря обновления (sweeper видит старую версию, обновляет статус, потом пользователь сохраняет – данные неконсистентны). Не указан порядок блокировок и приоритет (sweeper vs user). Нужен механизм optimistic locking или version check.

5. **Null confidence трактуется как 0 (секция 11.2)**: Для AI-inferred записей `confidence` обязательно, но для других – опционально. В conflict resolution и auto-approve `null` приравнивается к 0, что может привести к дискриминации старых записей, где поле не заполнено. Лучше требовать confidence для всех, или трактовать `null` как «неизвестно» и не участвовать в рейтинге.

6. **JSON Schema валидация при write, но отсутствует версионирование схем (секция 3.2)**: Если схема меняется (например, `feedback_v2`), старые записи остаются с `schema: "feedback_v1"`. Новый proposal может ссылаться на старую схему – как быть? Нет механизма миграции схем и проверки совместимости. Это может привести к ошибкам рендеринга в schema-driven UI.

7. **Backup и восстановление (секция 10.1)**: Упомянуто `_service/backup/`, но не описана процедура backup SQLite БД (WAL режим требует `VACUUM` или `checkpoint`). Нет гарантии point-in-time recovery. При потере SQLite все метаданные и версии пропадают (файлы на диске остаются, но без мета – бесполезны). Нужно обязательное резервное копирование с транзакционной консистентностью.

8. **Уровень 3.b: Service folders – конфликт с SQLite (секция 3.b)**: SQLite файл `aim_fs.db` размещён внутри `_service/db/`. Это нарушает принцип: вся служебная информация должна быть в SQLite, но сам файл SQLite – тоже служебный. При перемещении `_service/` на другой диск (например, SSD) ссылки внутри БД на файлы (пути) станут невалидны. Нужно использовать относительные пути или content-addressable store.

## MINOR ISSUES

- **Описание `on_dep_expire` (секция 3.1)**: Значение `"deprecate"` должно быть `"deprecate"` – исправлено в v6, но в тексте осталось `"deprecate"` без пояснения, что это опечатка.
- **Двойное хранение метаданных (секция 2)**: Файлы `_meta.jsonl` теперь дублируются в SQLite для обратной совместимости. Это удваивает объём хранения и требует синхронизации – не объяснено, что происходит при расхождении (кто primary). Лучше сделать SQLite источником истины с самого начала.
- **`max_inactivity_days` vs `max_absolute_pending_days` (секция 4.1)**: При offline >30 дней `auto_expire_at` не срабатывает, но `max_absolute_pending_days=90` – конфликт: через 90 дней proposal всё равно expire. Лучше объединить логику.
- **Batch-операции deadlock prevention (секция 4.2)**: Указан порядок возрастания ID. Но ID – ULID, который не гарантирует глобального порядка при параллельной вставке. Нужен фиксированный порядок по таблицам, а не по ID.
- **Пример auto-approve rules (секция 4.3)**: `auto_approve_observational_with_confidence_above: 0.9` – нет уточнения, что confidence >= или >. И не указано, что `null` confidence не допускается.
- **Search (секция 5.2)**: `LIKE` без escape для специальных символов – риск SQL injection? Не указан механизм параметризации (должен быть через `?`).
- **Dispute для >2 записей (секция 8)**: “интерфейс позволяет выбирать один вариант или комбинировать” – нет описания, как комбинировать в терминах графа (например, объединение contradictory фактов?).
- **Удаление пользователя отложено (секция 11.5)**: Если пользователь удаляется до Phase 2, его данные остаются в SQLite и на диске – GDPR compliance нарушен.

## STRONG POINTS

- **Использование SQLite WAL с транзакциями и SELECT FOR UPDATE** – индустриально правильный шаг, устраняет проблемы flock на NFS.
- **Чёткая обработка `null` confidence в conflict resolution** (tiebreaker по created_at) – простое и детерминированное решение.
- **Интеграция idempotency в транзакцию SQLite** – обеспечивает атомарность.
- **Отдельные метрики для MVP (p95 approve latency, throughput, inbox pending)** – даёт измеримые цели.
- **Процесс peer review с changelog v5→v6** – показывает зрелость документации.

## SUGGESTED ADDITIONS / SIMPLIFICATIONS

1. **Multi-tenant isolation**: 
 - Добавить обязательное поле `tenant_id` во все таблицы и проверять его в каждом SQL через `WHERE tenant_id = ?`.
 - Рассмотреть отдельную SQLite БД на пользователя (сложнее в backup, но проще изоляция). Или использовать SQLite ATTACH DATABASE для разных namespace.
 - Указать механизм шифрования метаданных (например, шифровать поле `scope` с помощью per-user ключа).

2. **Производительность**: 
 - Заменить единую БД на партиционирование по `user_id` (`aim_fs_<hash(user_id)>.db`).
 - Добавить connection pool (например, `r2d2` для rusqlite).
 - Указать поддержку `PRAGMA journal_size_limit` и регулярный checkpoint для WAL.

3. **Idempotency**: 
 - Не удалять строку при timeout, а устанавливать статус `failed` и оставлять на время TTL. Клиент при retry получит `conflict` и должен повторно отправить с новым ключом.
 - Добавить watchdog: если транзакция взяла блокировку дольше 30с, убивать процесс.

4. **Decay sweeper**: 
 - Использовать optimistic locking: добавить `version` поле в сущности, sweeper проверяет version при обновлении. Если изменилась – пропустить.
 - Запускать sweeper по расписанию (например, каждые 10 минут) с ограничением на число обработанных записей за раз.

5. **Conflict resolution**: 
 - Уточнить, что `contradicts` может быть обнаружен не только при создании, но и при approve (если proposal ссылается на уже conflicted запись).

6. **Реализуемость**: 
 - Уменьшить Phase 1 до 7 дней, но убрать decay sweeper и conflict resolution (только CRUD + propose/approve). Добавить их в Phase 2-3.

7. **Backup**: 
 - Добавить команду `aim_fs backup` с вызовом `.backup` SQLite, сохранением дампа скалярных значений и копией файлов на диске.

8. **Schema registry**: 
 - Ввести `schema_version` в frontmatter, чтобы при схеме `feedback_v2` старые записи можно было рендерить через отдельный view.

## SUFFICIENCY VS CLAUDE

AIM_FS превосходит Claude по большинству заявленных осей (L1-3, L5, L7-15). Однако по двум осям преимущество не доказано:

- **L4 (Semantic search)**: в MVP поддерживается только поиск по frontmatter (без body), в то время как Claude Code может использовать Tantivy или SQLite FTS для полнотекстового поиска. Указано, что в Phase 4 будет превосходить, но на момент MVP – уступает.
- **L6 (Decay/staleness)**: ограниченная глубина декей (3 уровня) и отсутствие поддержки каскадного декей для ссылок, которые не являются `depends_on` (например, `refines`). Claude Code не имеет декей, но его отсутствие не даёт AIM_FS однозначного превосходства – сравнимо.

Также не указана возможность **replay** событий (L12 в таблице Claude – "replay" отсутствует? В таблице только 15 осей, но replay не упомянут). Если replay – это воспроизведение действий пользователя, то в spec есть event log, но нет API для воспроизведения. Следовательно, преимущество по replay не подтверждено.

Вывод: spec требует доработки по изоляции, масштабированию и реализуемости, но концептуально верен. После исправления critical issues – ACCEPT.
