## CRITICAL ISSUES

1. **Race condition между decay sweeper и approval flow** (раздел 7 + 4.2) 
 - Фоновый sweeper (decay.rs) передвигает `expired` факты. Если в это же время происходит approve предложения, которое зависит от этого факта, sweeper может удалить/изменить состояние в момент, когда `atomic.rs` ещё не завершил commit. 
 - **Следствие:** approve может завершиться успехом, но факт тут же станет `deprecated`. Нет гарантии, что `decay.rs` использует тот же flock или проверяет блокировку перед мутацией. 
 - **Требование:** все мутации состояния (включая decay, conflict resolution timeout) должны проходить через единый lock `_service/.lock` (или per‑shard lock). В spec это не зафиксировано.

2. **Conflict resolution — потеря уже активных фактов** (раздел 8, пункт 5) 
 - Цитата: *«оба факта автоматически помечаются `superseded`»* при таймауте 7 дней. Если один из фактов уже был `active` (например, ранее approved), он становится `superseded` без участия пользователя. 
 - **Проблема:** пользователь мог полагаться на активный факт. Автоматическое supersede может удалить важное решение. 
 - **Решение:** при таймауте dispute следует помечать оба как `unresolved` (или `stale`), но не трогать статус активного факта. Предложение (если было) должно перейти в `expired`, а старый факт остаться `active`.

3. **Неатомарность INDEX при одновременных аппрувах** (раздел 5.3 + 4.2) 
 - В 5.3: *«Любое событие из `events/` → немедленное обновление соответствующего шарда в INDEX.json»*. 
 - Если два concurrent approve (каждый под своим flock на `_service/.lock`) попытаются обновить один и тот же шард, второй может прочитать устаревшую версию INDEX, перезаписать изменения первого. 
 - **Проблема:** rename атомарен только для конечного файла, но подготовка обеих версий может конфликтовать. Нужно блокировать шард на время read‑modify‑write. Либо агрегировать все изменения одного scope в один commit.

4. **Отсутствие каскадного decay для зависимых фактов** (раздел 7, 3.1) 
 - В frontmatter есть поле `links.depends_on`. Если факт A зависит от B, и B истекает по TTL, A должен стать `deprecated` или `stale`. 
 - Спецификация этого не регламентирует. Без каскадного decay пользователь может полагаться на устаревший вывод. 
 - **Требование:** добавить в decay sweeper рекурсивную проверку dependants. Либо указывать в schema правило: `on_dep_expire = "deprecate" | "keep"`.

5. **Scalability: flock на весь scope — bottleneck** (раздел 4.2) 
 - Цитата: *«Захват файловой блокировки на весь scope (user/global)»*. 
 - Для одного врача с 100 concurrent approve/reject операций (через несколько вкладок) это даст serialization. p95 < 500ms при 100 concurrent users — невыполнимо, если каждый ждёт flock. 
 - **Рекомендация:** перейти на per‑entity lock (например, `flock` на файл предложения или per‑project). Глобальная блокировка только для операций, затрагивающих несколько сущностей (например, merge dispute).

6. **Обновление identity.yaml при смене паспортных данных** (раздел 2, Tier 1) 
 - `identity.yaml` объявлен immutable. Как врач меняет имя или ORCID? Через отдельный proposal? Тогда это уже не immutable. 
 - **Проблема:** не описан workflow. Либо нужно разрешить overwrite через специальный «change identity» proposal, но тогда теряется смысл immutable.

7. **Multi‑tenant encryption: no key rotation, no revocation** (раздел 9) 
 - PII шифруется per‑user ключом, хранящимся в `.key.enc` (зашифрован мастер‑ключом). Если мастер‑ключ скомпрометирован — все данные расшифрованы. 
 - Нет процедуры ротации ключей, нет revoke доступа для пользователя (если он покинул организацию). 
 - **Требование:** минимум описать механизм re‑encryption при смене ключа и audit‑log доступа к ключам.

8. **Поиск MVP: full scan 50MB за 100ms — unrealistic на практике** (раздел 5.2) 
 - ripgrep может читать ~500 MB/s на SSD, но при 10⁵ записей, разбросанных по папкам и шардам, чтение `frontmatter` и `body` потребует много случайных seek. 
 - К тому же поиск по `rg` подразумевает вызов внешнего процесса — это latency overhead. 
 - **Следствие:** p95 latency approve может быть нарушена, если search вызывается для проверки дубликатов перед proposal. Нужен либо Tantivy с рождения, либо более точная оценка.

9. **Auto‑approve confidence — эвристика не определена** (раздел 4.3, 12.4) 
 - Цитата: *«confidence = heuristic(rarity, repetition, user_explicitness)»*. Это не спецификация. 
 - Без чёткой формулы auto‑approve может либо пропускать мусор, либо требовать ручного подтверждения для тривиальных фактов. 
 - **Требование:** привести хотя бы алгоритм (например, `confidence = sigmoid(repetition_count / threshold + user_match_score)`). Или сделать порог настраиваемым.

## MINOR ISSUES

- **YAML vs TOML:** в 3.1 frontmatter YAML, в 4.3 auto‑approve rules TOML — несогласованность. Рекомендую единый формат (YAML).
- **Naming:** `created_by.source` — `"user_message"` и `"ai_inferred"`, но в 4.1 auto‑approve для `user_message` без ограничений. Лучше `source: "user_explicit"`.
- **Опечатки:** «_meta.jsonl» везде, но в списке файлов уровня 2 написано `links.jsonl` и `events.jsonl` — нужно `_meta/links.jsonl`.
- **ULID vs UUID:** в 3.1 `id` — ULID? В тексте сказано ULID, но пример показывает формат `01HZAB12CDEF34GH56JK78MN90` — это ULID (Crocker). Хорошо.
- **Schema registry:** не описан процесс валидации при записи (в 10.1 есть `schema.rs`, но не указано, что делать при невалидном schema — reject proposal?).
- **Migration (10.3):** написано «без изменений», но в v1 не было деталей. Непонятно, как переносить legacy проекты.

## STRONG POINTS

- **Approval flow с атомарным commit** (prepare/rename) — корректное решение для single‑node.
- **Conflict resolution с таймаутом и blocked proposals** — проработано лучше, чем в большинстве AI‑memory систем.
- **Lazy sharded index** — разумный компромисс между производительностью и сложностью для MVP.
- **Schema‑driven UI** (L14) — потенциально сильное отличие от Claude, если будет реализовано (пока только упоминание).

## SUGGESTED ADDITIONS / SIMPLIFICATIONS

1. **Вырезать CAS и GC** — заменить на git‑подобное хранение с `git add` + `git commit` для мета. GC не нужен, пока размеры не превысят 1GB. Сэкономит 1–2 дня разработки.
2. **Добавить per‑entity lock** вместо глобального flock. Например, `flock` на файл `.lock` в папке проекта или пациента.
3. **Убрать auto‑expire с supersede из conflict resolution** — оставить только `unresolved` статус.
4. **Явно прописать каскадный decay** в раздел 7 и в `schema.rs` как `on_dep_expire`.
5. **Снизить амбиции Phase 1 до 5–6 дней** — добавить ещё 2 дня на тестирование race‑conditions.
6. **Добавить описание confidence heuristic** как формулу с весами, хотя бы в открытых вопросах.
7. **Для поиска MVP использовать in‑memory мульти‑map** (например, `HashMap<token, vec<id>>`) для frontmatter полей, а full‑scan для body. Это избавит от вызова `rg`.
