## CRITICAL ISSUES

1. **Ненадёжность `flock` на распределённых ФС (секция 4.2).** 
 Спецификация признаёт, что `flock` не гарантирован на NFS/FUSE, но откладывает переход на SQLite до Phase 6. Для медицинского применения это неприемлемо: любое использование облачных хранилищ или сетевых дисков приведёт к потере консистентности. **Требуется:** реализовать блокировку через O_EXCL с advisory‑файлами (что тоже ненадёжно) или немедленно перейти на SQLite WAL в MVP.

2. **Риск частичного выполнения при idempotency (секция 10.1).** 
 Если Rust‑процесс падает после записи `{"status": "processing"}`, но до завершения операции, таймаут 30s удалит запись, и повторный запрос начнёт операцию заново. Однако часть изменений (например, запись в events во временной папке) может уже существовать. Хотя rename не выполнен, возможно дублирование при повторной попытке с другим tmp‑именем. **Требуется:** документировать гарантии атомарности при краше; рассмотреть двухфазный протокол с журналом `_service/idempotency/` как единственной точкой фиксации.

3. **Null safety в conflict resolution (секция 8, п.5).** 
 При таймауте dispute для N≥2 выбор записи с наивысшим `confidence` не определён для случая, когда `confidence` не задан (null). Равенство confidence при null ведёт к неоднозначности. **Требуется:** явно указать, что null трактуется как 0, или использовать дополнительное поле `version_created_at` как tie-breaker в явном виде.

4. **Каскадный undo без транзакционной защиты (секция 4.3).** 
 Отмена auto-approved записи, на которую ссылаются другие активные факты (через `depends_on`), приводит к каскадному superseding всех зависимых записей. Если одна из зависимых записей уже участвует в dispute или используется в другой цепочке, каскад может создать неконсистентное состояние (например, disputed запись получит статус superseded, нарушая логику dispute). **Требуется:** ограничить undo записями без исходящих dependents (простая модель) или ввести двухфазный commit для каскада с откатом при невозможности supersede всех зависимостей.

5. **LLM‑мусор в observational data (секция 4, п.1).** 
 Данные текущего визита (intake, CBC) auto-approve без проверки, т.к. считаются "observational". Для медицинского применения это опасно: модель может сгенерировать несуществующие показатели. **Требуется:** хотя бы флаг `requires_verification` для таких записей, либо минимальная эвристика (отклонять при confidence < 0.9) с fallback в inbox.

6. **Неопределённость wildcard‑поиска (секция 5.1).** 
 `global_user_projects_index` кэшируется лениво, но не описана инвалидация при добавлении/удалении проектов. При активной работе пользователя кэш будет устаревать, приводя к пропуску новых записей. **Требуется:** чётко определить триггер инвалидации (event из events.jsonl) или перестроение при каждом таком запросе (с соответствующей оценкой производительности).

7. **Decay sweeper с ограниченной глубиной 3 (секция 7).** 
 При цепочке зависимостей глубже 3 уровней (A → B → C → D → E) истечение E обработает только A, B, C; D и остальные останутся без каскада. Это неполная обработка устаревания. Кроме того, циклы приводят к тотальному `deprecated`, что может удалить легитимно замкнутые факты (например, взаимные ссылки). **Требуется:** либо перейти к топологической сортировке полного графа (с отбрасыванием глубоких уровней по настраиваемому лимиту, но с явным логгированием пропущенных), либо для циклов выделить SCC и позволить пользователю решать, а не автоматически деприкейтить.

8. **Завышенная оценка реализуемости (секция 13).** 
 Phase 1 включает 10 пунктов (атомарность, frontmatter, propose/approve, events, index, CAS, full scan, idempotency, cleanup, race‑тесты) и оценивается в 10 дней. Это минимум 15–20 рабочих дней для одной опытной команды. Суммарно все фазы ~22 дня, не 7–10. **Требуется:** сократить MVP до минимума (профиль + проекты + atomic записи + базовый inbox), отложив CAS, decay, conflict, sharded index, full‑text search.

9. **Отсутствие единой транзакции для approve с обновлением нескольких сущностей (секция 4.2).** 
 Процедура approve изменяет как статус proposal в inbox, так и актуальное содержимое сущности (например, факт в profile). Хотя per-entity locks и rename tmp обеспечивают атомарность каждой части, нет гарантии, что оба изменения произойдут вместе при сбое между ними. **Требуется:** описать двухфазную фиксацию или использовать одну временную папку для всех изменений транзакции с единым rename корневой директории.

## MINOR ISSUES

- **Непоследовательность auto-approve user_command:** в разделе 4 сказано, что `source: "user_command"` auto-approve, но в секции 1.2 Tier 1 – только для profile при user_command. В секции 4.1 говорится, что все user_command auto-approve. Уточнить scope.
- **Опечатка `on_dep_expire: "deprecate"` (секция 3.1).** Должно быть `"deprecate"` → `"deprecate"`? Если опечатка, использовать `"deprecate"` как в статусе `deprecated`.
- **Дублирование `_meta/links.jsonl` и `_meta/events.jsonl` в проектах (секция 2).** Для однотипности можно вынести общую структуру `_meta/`.
- **Не описана очистка `_service/inbox/` после удаления proposal из pending.** После approve/reject/expire файл proposal перемещается в `approved/` или `rejected/` – это нормально, но не указано, что происходит с метаданными в events (дублирование).
- **Heartbeat только от Rust к Phoenix (секция 10.2).** Если Port зависает, Phoenix перезапускает процесс. Но нет механизма, чтобы Phoenix сообщал Rust о перезапуске – это нормально, но не описана обработка потерянных блокировок при убийстве процесса.
- **Формат `_service/idempotency/` – JSON?** Упомянут `<key>.json` – хорошо, но не указан полный формат записи. Стоит добавить пример.

## STRONG POINTS

- Идемпотентность с состоянием "processing" и таймаутом – продуманный компромисс между безопасностью и сложностью.
- Conflict resolution с правилами при таймауте для N записей (хоть и с недочётами) – сильный элемент.
- Multi‑tenant encryption с key rotation и revocation описаны подробно и корректно.
- Schema‑driven UI с авто‑генерацией формы – демонстрирует преимущество над CLI.

## SUGGESTED ADDITIONS / SIMPLIFICATIONS

1. **Сократить MVP до 10 дней:** исключить CAS, decay sweeper, conflict resolution, full‑text body search. Оставить: профиль, проекты, atomic записи с frontmatter, events log, inbox с per‑entity lock. Перенести остальное в Phase 2+. 
2. **Заменить `flock` на SQLite WAL для метаданных** сразу в MVP – решит проблемы NFS и блокировок. Файлы на диске остаются, все индексы и блокировки – в SQLite. 
3. **Добавить механизм верификации observational data:** флаг `requires_verification` для записей, созданных LLM без явной команды пользователя, с отображением в inbox как элемент низкого приоритета. 
4. **Определить правило для null confidence:** считать 0 или требовать обязательного выставления при AI‑inferred записях. 
5. **Для undo – ограничить отмену только записями без исходящих `depends_on`**, иначе требовать явного подтверждения каждого каскада с показом графа глубиной 1. 
6. **Добавить операцию удаления пользователя** с процедурой очистки данных, отзыва ключа и архивации проектов. 
7. **Исправить таблицу превосходства (L4, L2, L6):** честно указать, что MVP по некоторым осям уступает или сравним, а превосходство будет достигнуто в поздних фазах.

## SUFFICIENCY VS CLAUDE

Спецификация декларирует превосходство по 15 осям (L1–L15). Однако:

- **L2 (граф):** MVP использует full scan links.jsonl. Если Claude имеет материализованный граф (неизвестно), то AIM_FS не превосходит, а уступает по скорости. 
- **L4 (поиск):** MVP – full scan тела; Claude Code использует Tantivy (или grep?). Неясно, кто быстрее. Только Phase 5 (Tantivy) даёт превосходство. 
- **L6 (decay):** Ограничение глубины 3 – слабее, чем полный каскад (если у Claude есть decay). 
- **L13 (replay):** Event log есть, но процедура replay не детализирована и потребует значительных усилий для реализации. 

Для осей L1, L3, L5, L7, L8, L9, L10, L11, L12, L14, L15 – преимущество обосновано (однако L10 conflict resolution имеет недочёты). **Рекомендуется:** пересмотреть таблицу, указав для L2, L4, L6 статус "в разработке" или "сравнимо".
