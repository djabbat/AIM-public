## CRITICAL ISSUES 

1. **Неатомарность batch-reject в Race Condition (раздел 4.1, 4.2)** 
 Спецификация описывает атомарность single approve. Но не описана ситуация, когда пользователь в UI LiveView нажимает «reject all» для нескольких pending элементов. Каждый reject — отдельный per-entity lock. Если между reject’ами вклинивается concurrent propose для той же сущности, возможен lost-update. 
 *Требуется:* описать транзакцию с захватом всех локов в порядке возрастания ID, иначе reject/approve пачки неатомарны.

2. **Decay sweeper может deadlock с approve (раздел 7)** 
 Sweeper захватывает per-entity lock на сущность A, планируя изменить статус. Если approve на A требует захвата lock на B, а sweeper уже держит B после захвата A, возникает deadlock (порядок захвата разный). В спецификации не указан порядок захвата локов для sweeper. 
 *Требуется:* sweeper должен захватывать locks в возрастающем порядке ID сущностей, как и approve/reject.

3. **Граф связей не гарантирует консистентность при конкурентном добавлении (раздел 6)** 
 `links.jsonl` обновляется в рамках той же атомарной операции (prepare/rename). Но если два concurrent proposal создают противоречащие ссылки (contradicts) на одну сущность, то:
 - Оба proposal могут пройти approve до того, как второй заметит конфликт (раздел 8: «Новая запись помечается contradicts» — но как, если она ещё не в системе?).
 - Механизм обнаружения конфликта не описан: требуется ли full scan links.jsonl всех сущностей при каждом propose? Если да, производительность рухнет при 10⁵ записей.
 *Требуется:* описать, как на этапе propose проверяются существующие `contradicts` (через глобальный мульти-индекс? Если нет, это дыра). Рекомендуется добавить materialized view для contradicting пар с per-shard lock.

4. **Каскадный decay может создать цикл или бесконечную рекурсию (раздел 7)** 
 «Рекурсия ограничена глубиной 5». Но в реальности цепочка может быть длиннее (факт зависит от A → B → C …). Ограничение глубины 5 является произвольным и может пропустить реальные каскады. 
 *Требуется:* либо доказать, что max глубина < 5 в типовом использовании, либо реализовать через топологическую сортировку с обнаружением циклов.

5. **Migrate Legacy проектов без учёта возможных дубликатов (раздел 10.3)** 
 При миграции legacy проектов создаются `_meta/links.jsonl` и `events.jsonl`. Если legacy проект уже содержит файлы с одинаковыми датами/авторами, возможны дублирующие записи. Механизм дедупликации не указан. 
 *Требуется:* описать процедуру dedup при миграции (например, по SHA содержимого).

6. **Auto-approve правила не учитывают случай, когда confidence heuristic даёт >0.95, но пользователь на самом деле не согласен (раздел 4.3)** 
 Формула `sigmoid(repetition_count / threshold + user_match_score)` может дать высокую confidence на основе повторений, если пользователь заставил AIM повторить один и тот же факт из-за ошибки. Нет механизма отката auto-approved записи, если пользователь позже заметил ошибку. 
 *Требуется:* добавить undo-механизм (запись superseded с reversion) или ограничить auto-approve только для `source == "user_explicit"`.

7. **Пер-юзер ключи шифрования не защищены от атаки на мастер-ключ (раздел 9)** 
 Мастер-ключ шифрует per-user ключи. Если мастер-ключ скомпрометирован, все per-user ключи могут быть расшифрованы. Описана ротация мастер-ключа, но не описана процедура перешифровки per-user ключей новым мастер-ключом при ротации — является ли это batch job с потенциальным окном уязвимости? 
 *Требуется:* уточнить механизм безопасной ротации (ключи шифрования per-user дважды зашифрованы?).

8. **Offline-режим пользователя не предусмотрен для inbox (раздел 4.1)** 
 При offline пользователя перед expiry `auto_expire_at` (7 дней) AIM отправляет уведомление. Но если пользователь offline более 7 дней, все pending автоматически expire. Для врача это может быть критично (потеря диагностических proposal). 
 *Требуется:* предусмотреть механизм продления таймаута при offline или настройку per-user максимальной задержки ответа.

9. **Отсутствует проверка согласованности INDEX после сбоя (раздел 5.3)** 
 Шарды INDEX обновляются через rename. Если система падает между завершением rename и записью в events.jsonl, INDEX может быть обновлён, а событие потеряно. При восстановлении несоответствие не обнаруживается. 
 *Требуется:* реализовать checksum всего INDEX.shard после обновления и проверку при загрузке (replay event log до последнего согласованного снапшота).

10. **Глобальный INDEX не может быть построен лениво при 10⁵ записей (раздел 5.1)** 
 Шарды загружаются по запросу. Но для поиска по глобальному scope (например, «найти все факты confidence > 0.9» без указания пользователя/проекта) требуется загрузить все шарды. Это может занять O(N) времени. 
 *Требуется:* либо ограничить глобальный поиск только по шардовому индексу frontmatter (который может быть построен агрегатно), либо добавить глобальный оверхэд в метриках.

## MINOR ISSUES 

- **Смешение понятий `source` (раздел 3.1):** `user_message` и `user_explicit` — различие нечеткое. Рекомендую заменить `user_explicit` на `user_command`, чтобы избежать путаницы с `user_message`. 
- **Отсутствие description для схемы `schema_v1` (раздел 3.2):** в примере frontmatter `schema: "feedback_v1"`, но `_schemas/feedback_v1.json` не описан. Нужно хотя бы перечислить обязательные поля для каждой схемы. 
- **Таймаут conflict resolution (раздел 8):** «7 дней» указано цифрой, но не пояснено, откуда взялось это значение. Лучше параметризовать в конфиге. 
- **CAS retention policy:** «последние `retain_versions` версий» — не указано, как быть с файлами, которые изменились, но не имеют версионной истории (например, `identity.yaml` при proposal). 
- **Грамматика:** `per se` → `per se` (но это несущественно). 
- **Имена модулей:** `conflict.rs` не выделен, хотя conflict resolution описан. Лучше выделить отдельный модуль, а не смешивать с `index.rs` или `event_log.rs`.

## STRONG POINTS 

- Чёткое разделение на три уровня и проработка scoping для multi-tenant. 
- Per-entity lock с порядком захвата (deadlock prevention) — хороший подход, хотя нужны доработки. 
- Подробное описание метрик и фаз реализации — позволяет оценить реализуемость (даже с завышенной оценкой). 
- Введение confidence heuristic для auto-approve — попытка баланса между безопасностью и автоматизацией.

## SUGGESTED ADDITIONS / SIMPLIFICATIONS 

1. **Упростить граф:** вместо `links.jsonl` per entity использовать один глобальный файл связей с per-shard locking? Это может упростить обнаружение конфликтов. Но добавит нагрузку на одни и те же записи. В текущей версии лучше оставить per-entity, но добавить materialized inverted index для `contradicts`. 
2. **Исключить custom CAS из MVP:** Дедержится предположение, что CAS нужен для версионирования. Можно упростить — хранить версии как `filename.sha256` в той же директории, без выделенного хранилища `_service/cas/`. Это упростит GC и retention. 
3. **Добавить возможность пропускать approval для AI-inferred выводов с high confidence, если пользователь подтвердил предыдущие 10 аналогичных.** 
4. **Заменить Elixir‑Port на NIF для операций write-path:** Port добавляет задержку на сериализацию JSON. Для approve latency < 500 ms это может быть критично. NIF с careful cleanup — надёжнее. 
5. **Добавить `async_approve` как background job для случаев, когда пользователь не ждёт немедленного ответа:** уменьшит нагрузку на lock при burst. 
6. **Уточнить, как живёт `decay` для фактов, которые были обновлены (новая версия).** Должен ли TTL обнуляться? Сейчас не описано.

## SUFFICIENCY VS CLAUDE 

| Ось | Превосходство доказано? | Комментарий |
|-----|-------------------------|-------------|
| L1 (Index) | **Нет** | MVP full scan тела не превосходит Claude, а sharded index — только если он действительно lazy (необходима реализация). |
| L2 (Граф) | **Да** (условно) | links.jsonl per entity — лучше, чем ничего, но без DuckDB при большом количестве (10⁵) производительность может быть хуже произвольного запроса Claude (который использует embedding search). |
| L3 (Версии) | **Да** | CAS + retention явно превосходит Claude. |
| L4 (Поиск) | **Нет** | MVP уступает Tantivy в Claude, Phase 5 заявлена, но не реализована. |
| L5 (Approval) | **Да** | Inbox + блокировки — явное превосходство. |
| L6 (Decay) | **Да** (условно) | Каскадный decay лучше, чем verify-on-use, но отсутствие примеров с циклическими связями ставит под сомнение корректность. |
| L7 (Provenance) | **Да** | Полный provenance превосходит Claude. |
| L8 (Scoping) | **Да** | Scoping превосходит глобальный namespace. |
| L9 (Schemas) | **Да** | JSON Schema + валидация — сильное превосходство. |
| L10 (Conflict) | **Да** (условно) | Таймаут без потери активных — улучшение, но discovery конфликтов не описано. |
| L11 (Multi-tenant) | **Да** | Key rotation и revocation — сильное превосходство. |
| L12 (Lazy index) | **Нет** | Выигрыш в памяти 10x — утверждение без benchmark, не доказано. |
| L13 (Audit) | **Да** | Event log + replay — превосходство. |
| L14 (Schema-driven UI) | **Нет** | Указано «в разработке», преимущество не реализовано. |
| L15 (Atomic) | **Да** | Prepare/commit + per-entity lock — превосходство. |

**Итого:** по 5 осям (L1, L4, L12, L14, частично L2) преимущество либо не доказано, либо недоступно в MVP. Для полноценного превосходства необходимо:
- L1: перейти к lazy sharding с benchmark.
- L4: описать переход на Tantivy ASAP.
- L12: предоставить измерение памяти.
- L14: реализовать schema-driven UI до конца.
- L2: либо реализовать DuckDB, либо доказать, что links.jsonl + LRU справляется с 10⁵ entities.
