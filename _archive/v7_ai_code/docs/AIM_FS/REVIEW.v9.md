## CRITICAL ISSUES
1. **Несоответствие заявленного превосходства над Claude** 
 В разделе 1.2 и заголовке написано «превосходит Claude Code по следующим осям», а таблица 12 показывает, что по L4 (поиск) MVP **уступает**, по L6 (decay) **сравнимо**. Требование на превосходство по **всем** осям не выполнено. 
 *Необходимо:* либо исправить текст на «превосходит по большинству осей», либо явно указать, что L4 и L6 будут улучшены в Phase 2, а в MVP они уступают/сравнимы.

2. **Шифрование PII и метаданных – не входит в план MVP, но заявлено как часть дизайна** 
 Раздел 9 описывает per‑user ключи, колоночное шифрование (AES‑256‑GCM), ключи в HSM/Vault. В плане реализации (раздел 13) в Phase 1 об этом ни слова. Если шифрование обязательно для соответствия требованиям безопасности, MVP не может быть построен за 7 дней. 
 *Необходимо:* явно переместить шифрование в Phase B или в Roadmap, а в MVP использовать только tenant_id и отсутствие шифрования на стороне приложения (SSD‑уровень – достаточно для одного доктора).

3. **Отсутствует таблица `versions` и описание версионирования** 
 В разделе 3.3 сказано, что версионирование в MVP реализовано через хранение истории изменений в SQLite (таблица `versions`). Однако в схеме таблиц (раздел 5.1) `versions` не упомянута. Непонятно, хранятся ли там только изменения frontmatter или также тела файлов, и как происходит запись/чтение. 
 *Необходимо:* добавить DDL `versions` (id, entity_id, data JSON, hash, created_at) и описать, что при каждом approve создаётся новая версия.

4. **Атомарность проверки конфликтов при propose не гарантирована** 
 Раздел 8: «Перед созданием proposal выполняется SELECT.. FROM entities WHERE id = ? FOR UPDATE, затем SELECT.. FROM links WHERE target_id = ? AND link_type = 'contradicts'». Второй запрос **не блокирует таблицу `links`**. Два concurrent propose могут одновременно увидеть отсутствие конфликта и вставить противоречащие записи. 
 *Необходимо:* либо использовать `BEGIN IMMEDIATE` (блокирует запись всей БД на время транзакции), либо блокировать таблицу `links` через `SELECT.. FROM links WHERE.. FOR UPDATE` (если SQLite поддерживает row‑level locking на virtual `rowid` – нет). Для MVP проще использовать `BEGIN IMMEDIATE` с маленькой задержкой.

## MINOR ISSUES
- **Опечатка `on_dep_expire`:** `"deprecate"` – не английское слово (возможно, имелось в виду `"deprecate"` или `"archive"`). 
- **Разнобой в планах на auto‑resolution:** раздел 8 говорит «отложен до Phase 2», а раздел 13 (Phase 4) обещает его через 2 дня. Синхронизировать: либо Phase 2, либо Phase 4. 
- **Frontmatter versions:** поле `versions` (список хешей) дублирует данные из таблицы `versions` и будет расходиться при ручном редактировании. Рекомендую оставить только последнюю версию в frontmatter, а историю хранить в SQLite. 
- **Scope с wildcard `*`** в MVP: для одного пользователя избыточно. Можно упростить до `scope: nil | List`. 
- **`tenant_id` = user_id**: если пользователь меняет username, все данные ломаются. Лучше использовать UUID для user_id, а не UNIX‑имя. 
- **`created_by.agent` дублирует `created_by.source`**: `"aim"` vs `"user_message"` – непонятно, как различать действия AIM и system. Можно убрать `agent`, оставить `source`. 
- **В разделе 4.1** `source: "user_command"` помечается как `auto_approve`, но событие пишется в лог – это conflict с `approval.auto_approve_user_commands` (раздел 4.3). Лучше явно сказать, что при `auto_approve_user_commands: true` такие записи сразу статус `active` без попадания в inbox. 
- **В разделе 10.1** `max_idempotency_processing_sec` = 30 секунд – для одного пользователя и локального SSD это слишком много; 5 секунд достаточно. 
- **Phase 1 (7 дней)** очень плотная; риск не уложиться. Рекомендую вынести conflict detection и sweeper в Phase 2, а в MVP оставить только CRUD + propose/approve/reject + inbox.
