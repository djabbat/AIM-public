## CRITICAL ISSUES

1. **Отсутствие DDL и описания таблицы `proposals`.** 
 Раздел 5.1 перечисляет таблицы (`entities`, `proposals`, `events`, `links`, `locks`, `idempotency`, `versions`), но DDL приведён только для `entities` и `versions`. 
 **Проблема:** без структуры `proposals` (статусы: `pending`, `approved`, `rejected`, `blocked`? поля: `entity_id`, `proposal_type`, `approved_by`, `created_at`, `tenant_id`) approval flow не может быть реализован однозначно. 
 **Требование:** добавить полный DDL для `proposals` и описание статусов, включая механизм `blocked` при конфликте (раздел 8).

2. **Неопределённость таблицы `locks` и её использования.** 
 В разделе 4.2 упоминается таблица `locks` в порядке захвата, но её назначение не раскрыто. При использовании `BEGIN IMMEDIATE` отдельная таблица для блокировок не нужна — она только усложняет архитектуру и может привести к deadlock-ам (например, если программа случайно смешивает блокировки на уровне приложения и SQLite). 
 **Требование:** убрать таблицу `locks` из MVP; если нужна для будущих оптимизаций, перенести в Roadmap с явным обоснованием.

3. **Примеры frontmatter используют `user_id` в виде имени вместо UUID.** 
 Раздел 3.1: `user_id: "djabbat_gmail_com"`. После исправления (changelog п.5) `tenant_id` и явно `user_id` должны быть UUID. Примеры вводят в заблуждение — разработчик может захардкодить не-UUID. 
 **Требование:** заменить на валидный UUID (например, `"a1b2c3d4-.."`) во всех примерах frontmatter и в разделе 9 (`users/<user_id>/`).

4. **Неточное описание поведения concurrent propose (раздел 11.5).** 
 Утверждается, что "Второй proposal получает `conflict`". На самом деле при `BEGIN IMMEDIATE` второй proposal будет **ждать** освобождения блокировки и выполнится нормально, если не возникнет конфликта на уровне данных (optimistic locking). Ошибка может привести к неправильной реализации retry‑логики. 
 **Требование:** скорректировать на: "Одновременные propose на разные сущности сериализуются через блокировку БД без ошибки; конфликт возникает только при modify одной сущности (через проверку `version`)."

5. **Противоречие в таблице превосходства для L10.** 
 В таблице 12 указано "MVP превосходит", а в скобках "(сравнимо с Claude, но мы явно обрабатываем)". Это не соответствует заявлению о 13/15 осей превосходства и может запутать читателя. 
 **Требование:** унифицировать — либо "превосходит" (с пояснением почему), либо перевести в категорию "сравнимо" и скорректировать общее количество.

6. **Не описан workflow при `reject` proposal.** 
 В разделе 4 описывается только `approve` и `reject`, но не сказано, что происходит с entity после reject (статус остаётся неизменным? proposal удаляется? сохраняется история?). Аналогично не описан статус proposal после reject. Дыра в полноте. 
 **Требование:** добавить явное описание: после reject proposal получает статус `rejected`, entity не изменяется; proposal остаётся в базе для аудита.

## MINOR ISSUES

- **Опечатка в значении `on_dep_expire`:** написано `"deprecate"` (раздел 3.1). Должно быть `"deprecate"` (или `"deprecate"`? Нестандартное слово). Лучше `"deprecate"`.
- **Отсутствие индекса по `(tenant_id, status)` в таблице `entities`:** хотя упомянут composite index (tenant_id, status) — есть. Но нет отдельного индекса по `(status)` для sweeper, который сканирует по `where status = 'active' and expires_at <..`. Это может быть незначительно, но для 10⁴ записей не критично.
- **Раздел 11.1 (LLM‑мусор) ссылается на v8 без копирования текста.** Для самодостаточности spec стоит привести краткое описание фильтрации (confidence threshold, manual review). Иначе читатель вынужден искать v8.
- **В DDL для `versions` не указан внешний ключ с `ON DELETE CASCADE`.** При удалении entity версии остаются «сиротами». Для MVP возможно, но стоит оговорить политику.
- **Раздел 5.1: упоминается `cache_size=-8000` (8 MB).** Для MVP с одним пользователем нормально, но лучше явно указать, что это временное решение для демонстрации.

## STRONG POINTS

- **Idempotency с таймаутом и очисткой при старте** — надёжный подход для распределённого окружения, корректно адаптированный под один процесс.
- **Использование `BEGIN IMMEDIATE` + optimistic locking** даёт гарантию serializable изоляции для запросов modify без сложных схем блокировок.
- **Автоодобрение с confidence heuristic** и отдельный inbox для observational data — учитывают реалии AI-агента.
- **Чёткое разделение на Phase 1–6 с Roadmap** позволяет избежать over-engineering в MVP.
- **SMART-метрики** конкретны и реалистичны для одного пользователя.
